# 3D検証ビューア (three.js)

STL / STEP / STP / OBJ / 3MF を**無劣化**（メッシュをそのまま）でブラウザ表示し、
3Dデータの検証に使うためのWebアプリケーション。
**G-code（Bambu/Orca）のツールパスも読み込んで、スライス結果を3Dで検証できる。**

公開・継続開発向けに **Svelte 5 + Vite 8 + TypeScript** を採用している。Three.jsのシーン管理・
各形式のパーサーは命令的なコアとして維持し、通知とモデル一覧はSvelteコンポーネント／ストアで描画する。
読み込み失敗や表示省略はブラウザ標準のダイアログを止めず、画面右上の閉じられるトーストで表示する。

## 起動

Node.js 20.19以上を用意する。`file://` 直開きではなく、ViteのローカルサーバまたはHTTPSの
静的ホスティングで利用する。

```bash
npm install
npm run dev
# → ブラウザで http://localhost:8000 を開く
```

> `uv run python -m http.server` や他の静的サーバーでは起動しない。`.ts` の変換と
> `three` などの依存解決をViteが行うため、必ず `npm run dev` を使う。

起動したら「ファイルを開く」かウィンドウへドラッグ&ドロップ。手元の `.stl` / `.step` /
`.3mf` / `.gcode` などを投入する。

## 本番ビルド

```bash
npm run check
npm run build
# dist/ をHTTPS対応の静的ホスティングへ配置
```

`dist/` は最適化済みの静的ファイルなので、GitHub Pages・Cloudflare Pages・Netlifyなどにそのまま
配置できる。`npm run preview` は配布前のローカル確認用。

主な読み込みは「ファイルを開く」「フォルダーを開く」とドラッグ&ドロップ。どのブラウザでも
（公開HTTPS上でも）動き、サーバ側処理は一切ない。

さらに **Chrome / Edge** では「選択フォルダーを監視（自動更新）」ボタンが出る。押すとフォルダー
選択画面が開き、ブラウザが読み取り権限を保持して1秒ごとに差分を確認するため、ファイルの
追加・更新・削除が自動でビューアへ反映される（File System Access API。`localhost` または
HTTPSが要件）。**Safari / Firefox は同APIに未対応なので、このボタンは自動的に非表示**になり、
「フォルダーを開く」での一回読み込みに誘導される。

### G-code（スライス結果）の表示

bambu-slice スキルが吐く `out/slice/plate_1.gcode` をそのまま投入するとツールパスを描画する。
`result.json` を一緒に投入すると印刷時間・フィラメント量・フィーチャ別時間配分・警告も表示。

**Bambu Studio の `*.gcode.3mf`（スライス済みプロジェクト）もそのまま投入できる。**
中の `Metadata/plate_*.gcode` を取り出してツールパス表示し、`slice_info.config` から
予測時間・フィラメント重量・警告（bed温度過多等）も拾う。
※Bambuの `.gcode.3mf` はメッシュを含まないことが多く（`3dmodel.model` が空）、その場合は
ツールパスのみ表示。メッシュ入りの通常 `.3mf` は従来どおりメッシュ表示にフォールバックする。

同一オリジンで公開しているファイルは URL パラメータで自動ロードできる（`result.json` は同ディレクトリを自動取得）:

```
http://localhost:8000/?gcode=/samples/plate_1.gcode
# STLと重ねたい場合（オーバーレイ）:
http://localhost:8000/?model=/samples/model.stl&gcode=/samples/plate_1.gcode
```

### フォルダーの一括表示・自動更新

フォルダー選択は対応形式を再帰的にまとめて読み込み、追加・更新・削除を自動反映する。複数の
フォルダーを選べ、サイドバーに**監視中フォルダー一覧**（フォルダー名・件数）を表示する。各行の
「監視解除」ボタンで個別に監視を止められ、そのフォルダー由来のモデルはまとめて撤去される。
初期状態は重ならないグリッド配置で、「レイアウト」と「モデル間隔」で比較用に調整できる。
監視は Chrome / Edge のみ。「フォルダーを開く」はどのブラウザでも使える一回読み込み。

## フロントエンド構成

- `src/lib/ToastStack.svelte`: エラー・警告・情報の通知キュー
- `src/lib/ModelList.svelte`: モデル一覧、表示切替、削除、色変更、G-code選択
- `src/lib/*.ts`: SvelteストアとThree.jsコアのイベント境界
- `index.html`: Three.jsのシーン、メッシュ/G-codeのパース、レンダリング。設定パネルは今後この境界を
  維持したままSvelteコンポーネントへ段階的に移す

> ドラッグ&ドロップで result.json を効かせるには、gcode と result.json を**同時に**投入する
> （ブラウザのファイル入力は兄弟ファイルを自動では読めないため）。

## 形式ごとの読み込み

| 形式 | 方法 | 備考 |
|------|------|------|
| STL  | three.js `STLLoader` | バイナリ/ASCII両対応・無劣化 |
| OBJ  | three.js `OBJLoader`  | メッシュ連結 |
| 3MF  | three.js `3MFLoader`  | メッシュ連結 |
| STEP/STP | フル版 OCCT `opencascade.js` (WASM) をブラウザ内実行 | **面ごとの色まで保持**。初回のみCADエンジン(wasm 約48MB/gzip~13MB)をCDN取得・以降キャッシュ。XCAF経由で読み色付きglTFをメモリ生成→`GLTFLoader`で表示。mm単位維持 |
| glTF/GLB | three.js `GLTFLoader` | マテリアル色・頂点色を保持して表示（単一ジオメトリへ統合し頂点カラーに焼き込む） |

> STL/OBJ/3MF は元メッシュをそのまま描画（頂点・三角形を改変しない）。
> STEP はB-rep（厳密曲面）なので、表示には三角形分割が必須。寸法・形状検証には十分だが、
> 三角形数は変換時のテッセレーション精度に依存する。

### STEP の色について

KiCad/CadQuery 由来 STEP の色は**面ごとの上書き**（`OVER_RIDING_STYLED_ITEM` →
`ADVANCED_FACE`）で付くため、軽量な `occt-import-js` では読めずほぼ単色になる。
そこでビューアは**フル版 OCCT (`opencascade.js`) をブラウザ内で動かして**全色を解決する
（imagetostl 等がサーバ側でやっている STEP→色付き glTF 変換を、クライアントだけで実行）。
初回の STEP 読み込み時に CADエンジン(wasm)を 1 回ダウンロードする（以降キャッシュ）。

### バッチ/事前変換 → `step2glb.mjs`（任意）

同じ変換を Node CLI でも実行でき、`.glb` を事前生成できる（大量ファイルや配布用）。
ビューアに直接 STEP をドロップすれば不要だが、ブラウザ内変換を待ちたくない場合に。

```bash
npm install                 # 初回のみ（opencascade.js を取得）
node step2glb.mjs <input.step> [output.glb]
```

- glTF は既定でメートル単位だが、入出力単位を mm に揃えるので寸法・体積はそのまま。
- `.glb` は再生成可能な巨大成果物なので gitignore 済み。

## 検証機能

- **メッシュ情報**: 三角形数 / 頂点数 / XYZ寸法(mm) / 体積近似(cm³)
- **ワイヤーフレーム / エッジ(折れ線)**: 面構成・特徴稜線の確認
- **法線の向きで色分け** (`MeshNormalMaterial`): 面の方向チェック
- **裏面を赤で警告**: 反転した法線・裏返り面の検出（背面が赤く見える）
- **断面 (Zクリップ)**: 高さスライダで内部構造（ジャイロイド等）を確認、反転可
- **バウンディングbox / グリッド / 軸**: 赤=X 緑=Y 青=Z
- **視点プリセット**: ISO / 前 / 上 / 右 / フィット
- **背景・モデル色**切替、自動回転
- **配置**: 重ねる／横並び／グリッド配置。複数の案を同時に比較するときに重なりを回避できる
- **選択・ラベル**: キャンバス上のモデルまたは一覧名を選択すると、一覧カードと3Dの枠を青く強調。
  モデル上部には小さな名前ラベルを表示し、不要なら「モデル名ラベル」で隠せる
- **定規**: 「2点間を計測」を有効にし、メッシュ表面を2点クリック。距離・ΔX/ΔY/ΔZ・水平距離をmmで表示する

### G-code検証

- **フィーチャ別カラー**: Outer/Inner wall・Sparse/Solid infill・Top/Bottom surface・Bridge・
  Support・Skirt などを色分け（bambu-slice の凡例に概ね一致）。凡例クリックで個別表示切替。
- **レイヤースクラブ**: スライダで「下から第Nレイヤーまで」を表示。造形の積み上がりを追える。
- **移動(トラベル)線**: 非押し出し移動の表示切替（既定オフ）。
- **メッシュを半透明ゴースト化**: 元STLを薄く重ね、ツールパスとオーバーレイ。
  **薄壁がスライサーで消えていないか**（ノズル幅未満で壁が出ない層）を3Dで直接確認できる。
- **統計HUD**（result.json があるとき）: 予測印刷時間・フィラメント長/重量・層厚・壁数・
  infill密度・`warning_message`（floating regions 等）・フィーチャ別の時間配分バー。
- STLと同じ正規化（XY中心・底面z=0）で配置するので、同じ造形物ならオーバーレイが揃う
  （中心合わせ用bboxからは Skirt/Brim を除外）。

#### 色分けモード（フィーチャ / オーバーハング / 体積流量）

- **フィーチャ別**: 既定。Outer/Inner wall・infill 等で色分け。
- **オーバーハング**: 各押し出しセグメントの真下（直下レイヤー）に材料があるかを占有グリッド
  （0.8mm）で判定し、**支えの無いセグメントを赤**にする。サポート要否・TPUの垂れ懸念箇所を
  数値に頼らず3Dで把握できる（第0層はベッド支持として緑）。
- **体積流量(mm³/s)**: `E長 × フィラメント断面積(π·0.875²) × 速度(F/60) ÷ セグメント長` を計算し
  青→緑→黄→赤でマップ。**上限(mm³/s)を超えるセグメントは赤**で警告。TPUは最大流量が低いので
  上限にTPUの値を入れると吐出不足の危険箇所を事前に見つけられる。実測ピークも表示。

#### ビルドアップ再生・同期・消えた壁検出

- **▶再生**: レイヤーを下から順にアニメ再生（30〜240層/秒）。印刷の積み上がりを確認。
- **全gcodeのレイヤーを同期**: 2つ以上のgcodeを読み込んだとき、レイヤー操作・再生を全gcodeへ
  同期。既存の「横に並べて比較」と併用すれば、パラメータ変更前後を**左右同レイヤーで視覚比較**できる。
- **消えた壁を検出（現レイヤー・要STL）**: STLを一緒に読み込んでおくと、現在のレイヤー高さで
  「STLにはソリッドがあるのに gcode に押し出しが無い」XYセルを検出して**桃色の点**で表示。
  薄壁がノズル幅未満でスライサーに消された箇所の自動発見。レイヤーを動かすと各高さで再判定。
  ※メッシュをz平面でレイキャストして内外判定するため、オーバーレイ（横並べOFF）で使う。

## 注意

- 体積は watertight（閉じた多様体）前提の符号付き体積の絶対値。非多様体メッシュでは不正確。
- 大きなSTL（数百万三角形）は読み込みに時間がかかる。
